# ZEUES v5.0 — Single Page + Modal Stack

## Vision

Reemplazar el flujo multi-pantalla (9 páginas con navegación lineal) por una **pantalla única con cards de spools + modales apilados**. Simplifica la UX eliminando navegación innecesaria y centraliza todas las operaciones en una sola vista.

## Core Value

"Gestionar todos mis spools desde una sola pantalla, con operaciones a un click de distancia."

## Requirements

### CARD — Pantalla Principal (6 requisitos)

- **CARD-01**: Botón "Añadir Spool" abre modal con SpoolTable + filtros existentes
- **CARD-02**: Cards de spools añadidos muestran: TAG, operación actual, acción, worker, tiempo en estado
- **CARD-03**: Spools persisten en localStorage (solo tag_spool), estado se refresca del backend cada 30s
- **CARD-04**: MET APROBADA remueve spool del listado automáticamente
- **CARD-05**: MET RECHAZADA mantiene spool para ir a Reparación
- **CARD-06**: Se pueden añadir múltiples spools y operar cada uno individualmente

### MODAL — Flujo de Operaciones (8 requisitos)

- **MODAL-01**: Click en card → OperationModal (ARM/SOLD/REP/MET filtrado por estado del spool)
- **MODAL-02**: ARM/SOLD/REP → ActionModal (INICIAR/FINALIZAR/PAUSAR/CANCELAR filtrado por estado)
- **MODAL-03**: INICIAR/FINALIZAR/PAUSAR → WorkerModal (workers filtrados por rol de operación)
- **MODAL-04**: CANCELAR no requiere worker — vuelve a pantalla principal directo
- **MODAL-05**: MET → MetrologiaModal (APROBADA/RECHAZADA)
- **MODAL-06**: Al seleccionar worker o resultado MET, se ejecuta API call y vuelve a pantalla principal
- **MODAL-07**: NotificationToast muestra feedback éxito/error en pantalla principal al completar acción
- **MODAL-08**: Eliminamos selección de uniones — PAUSAR reemplaza completación parcial

### STATE — Lógica de Estado (6 requisitos)

- **STATE-01**: Operaciones válidas dependen del estado del spool:
  - Libre → ARM
  - ARM-PAUSADO → SOLD
  - ARM-COMPLETADO → SOLD
  - SOLD-COMPLETADO → MET
  - RECHAZADO → REP
  - PENDIENTE_METROLOGIA → MET
  - BLOQUEADO → ninguna
- **STATE-02**: Acciones válidas dependen del estado de ocupación:
  - Libre → INICIAR / CANCELAR
  - En progreso (ocupado) → FINALIZAR / PAUSAR / CANCELAR
- **STATE-03**: CANCELAR en spool libre (recién añadido) = quitar del listado (frontend-only, no API)
- **STATE-04**: CANCELAR en spool con operación activa = reset operación en backend + quitar del listado
- **STATE-05**: Timer muestra tiempo desde Fecha_Ocupacion solo cuando spool está ocupado
- **STATE-06**: Spools PAUSADOS muestran badge estático "PAUSADO" sin timer

### API — Endpoints Nuevos (3 requisitos)

- **API-01**: `GET /api/spool/{tag}/status` — estado individual con campos computados (operacion_actual, estado_trabajo)
- **API-02**: `POST /api/spools/batch-status` — refresh batch aceptando `{tags: ["TAG1", "TAG2"]}`
- **API-03**: ✅ Modificar `POST /api/v4/occupation/finalizar` — aceptar `action_override: 'PAUSAR' | 'COMPLETAR'` para eliminar dependencia de selección de uniones (completed 2026-03-10 in plan 00-03)

### UX — Experiencia (4 requisitos)

- **UX-01**: Modal "Añadir Spool" muestra spools ya añadidos como deshabilitados/grises
- **UX-02**: Notificaciones toast auto-dismiss (3-5 segundos)
- **UX-03**: No optimistic updates para transiciones de estado — usar loading spinner y esperar respuesta API
- **UX-04**: Mantener paleta visual Blueprint Industrial (navy #001F3F, naranja #FF6B35) y mobile-first

## Lifecycle de un Spool en el Listado

```
Añadir → ARM INICIAR(worker) → ARM FINALIZAR(worker) o ARM PAUSAR(worker)
  → SOLD INICIAR(worker) → SOLD FINALIZAR(worker) o SOLD PAUSAR(worker)
  → MET APROBADA → se libera del listado

Si MET RECHAZADA:
  → REP INICIAR(worker) → REP FINALIZAR(worker)
  → MET APROBADA → se libera

Max 3 ciclos REP antes de BLOQUEADO.
```

## Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| D-01 | Single page + modal stack (no routing) | Elimina navegación innecesaria, todo a un click |
| D-02 | localStorage para persistencia de tags | Simple, sin backend, persiste entre sesiones |
| D-03 | Polling 30s para refresh de cards | Balance entre actualidad y rate limits de Sheets |
| D-04 | Estado_Detalle como fuente de verdad para estado del card | Ya codifica toda la info necesaria, evita columnas nuevas |
| D-05 | action_override en FINALIZAR | Elimina selector de uniones sin romper lógica backend |
| D-06 | CANCELAR dual: frontend-only vs backend según ocupado_por | Resuelve ambigüedad sin endpoint adicional |
| D-07 | Timer solo con spool ocupado, badge estático en PAUSADO | Fecha_Ocupacion se borra al pausar |
| D-08 | Reutilizar componentes existentes (SpoolTable, Modal, etc.) | Menor esfuerzo, código probado |
| D-09 | No optimistic updates | Evita divergencia estado frontend/backend |
